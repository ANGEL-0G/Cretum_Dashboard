Workflow Altareturn → Reportes personalizados

Cómo vamos a usar los 1000+ PDFs de Altareturn para alimentar la base y generar reportes personalizados por LP, todo desde cretumdesk.com.

El objetivo final

Generar reportes personalizados como el de Nathaly Marcus (Anthropic + SpaceX, MOIC, escenarios IPO, dilución, eventos próximos) on-demand desde cretumdesk.com, en lugar de armarlos manualmente cada vez con Claude alimentado a mano. Los PDFs de Altareturn son el insumo; los reportes son el producto.

El problema en una línea

Altareturn tiene ~1000 PDFs heterogéneos (capital calls, K-1s, NAVs, distribuciones, todo mezclado) + Salesforce con datos estructurados, ambos sin API disponible. Hay que convertirlos en datos consultables desde cretumdesk.com sin pagar más de lo necesario — y encima poder usarlos para generar reportes personalizados por LP.

La buena noticia: ya tienes la mitad

Tu Supabase ya tiene las tablas investors, contacts, companies, series e investments con todos los datos numéricos del reporte (commitment, shares, valuaciones, MOIC). Lo que falta es el contexto cualitativo que vive en los PDFs (tesis, eventos destacados, próximos eventos, comentario). Por eso el reporte sale así de bien:

  • ~60% del reporte es cálculo determinístico desde tablas (panorama, métricas, escenarios IPO, comparativa)
  • ~40% del reporte es prosa / contexto extraído de PDFs vía LLM (tesis, eventos, perspectiva)

Esquema de la tabla documents

create table documents (
  id            uuid primary key default gen_random_uuid(),
  filename      text not null,
  source_path   text not null unique,    -- ruta en Dropbox
  file_hash     text not null,           -- SHA256, para idempotencia
  raw_text      text,                    -- texto extraído
  page_count    int,
  byte_size     int,
  extraction_method text,             -- 'pdf-parse' o 'tesseract'
  extracted_at  timestamptz default now()
);

-- Índice GIN para búsqueda full-text en español
create index documents_search on documents
  using gin (to_tsvector('spanish', raw_text));

4
Extracción estructurada automática
Cada PDF nuevo ~$1 backlog, < $1/mes ongoing

Aquí está la pieza crítica: cada vez que un PDF entra al sistema (sea del backlog o nuevo), Claude lo lee y rellena automáticamente las tablas estructuradas (investors, investments, etc.). Sin esto, los reportes nunca tendrían datos frescos cuando llega un LP nuevo o cuando un cliente existente recibe un capital account statement nuevo.

Cómo se integra con la Fase 1

Script de Fase 1: extrae raw_text del PDF (gratis, local)
↓
Inmediatamente después: manda raw_text a Claude Haiku con schema JSON: 
{
  "doc_type": "capital_account_statement" | "welcome_letter" | "k1" | ...,
  "investor_name": "Nathaly Marcus Roffe",
  "investor_email": "nmarcus@bienesta.com",
  "fund_name": "MVP Opportunity Fund VI LLC",
  "series": "23F",
  "company": "Anthropic",
  "commitment_usd": 50000,
  "shares": 1497,
  "pps_entry": 31.37,
  "as_of_date": "2026-05-13",
  "confidence": 0.92
}
↓
Backend:
  • Si confidence ≥ 0.85 y los campos hacen match → upsert automático a las tablas
  • Si confidence < 0.85 o hay datos ambiguos → cola de revisión humana
  • Guarda link documents.investor_id = match encontrado (resuelve "qué docs son de quién")

Por qué esto es importante para los reportes

Cuando alguien pide el reporte de Nathaly Marcus, el sistema necesita saber automáticamente:

  • Qué investor_id es ella (matcheado al nombre del PDF)
  • Qué documentos son suyos (linkeados por documents.investor_id)
  • Qué shares tiene actualmente (poblado por extracción de los capital account statements)

Sin la Fase 4, todo eso sería taggeo manual.

4.5
Monitoreo de valuaciones (empresas en portafolio)
Diario / semanal ~$1-3/mes

Las valuaciones de las empresas privadas no cambian diario en realidad — cambian cuando hay un evento (ronda, secondary, S-1, noticia). El sistema busca diariamente si hubo un evento y, si lo hay, propone una actualización para revisión humana.

📌 Aclaración importante sobre "valuaciones que cambian diario"
Para empresas públicas (acciones que cotizan), sí cambian diario y se puede jalar de Yahoo Finance gratis. Para empresas privadas (Anthropic, SpaceX, todas las del portafolio MVP), las valuaciones se mueven en saltos discretos cuando hay rondas o secondary sales. "Diario" para estas no aplica — lo que sí podemos hacer diario es buscar noticias.

Cómo se ve

Cron diario (ej. 7 AM CDMX) en Vercel
↓
Para cada empresa en companies donde is_public = false:
  • Claude con web search: "noticias últimos 7 días sobre [empresa] valuación / ronda / secondary"
  • Si encuentra algo material → crear registro en tabla valuation_events
  • Si no → log silencioso, sigue al siguiente
↓
Para cada empresa donde is_public = true:
  • Pulla precio actual de Yahoo Finance (gratis, sin API key)
  • Update directo en companies.current_price
↓
UI en cretumdesk.com → "Eventos de Valuación" muestra los hallazgos:
  • Evento detectado, fuente, snippet de la noticia, valuación propuesta
  • Botones: "Aplicar a investments" / "Ignorar" / "Editar valor antes de aplicar"

Por qué no automático sin revisión humana

Una noticia puede decir "SpaceX podría valuarse en $2T en su próximo round" — eso no significa que el mark cambió, es especulación. Otra puede decir "Forge muestra acciones de Anthropic transándose a 50% premium" — eso sí podría ser un re-mark. Claude no debería tomar esa decisión solo, va a revisión. Pero el trabajo pesado (buscar, leer, resumir) ya está hecho.

Fuentes que vamos a usar

  • Empresas privadas: Claude con web search → TechCrunch, The Information, Bloomberg, WSJ, Reuters, PitchBook (lo que sea público).
  • Secondary markets (cuando estén públicos): Forge Global, EquityZen — feed RSS/news si lo tienen.
  • Empresas públicas: Yahoo Finance (gratis, sin API key, vía scraping ligero o librerías como yahoo-finance2).

Modelo de generación: on-demand con caché inteligente

Cada reporte se genera solo cuando alguien lo pide y se cachea por LP. Si los datos de ese LP no han cambiado desde la última generación, sirve la versión cacheada al instante ($0). Cuando algo cambia (PDF nuevo, valuación, edición manual), el caché se invalida y la próxima petición regenera con datos frescos.

¿Qué invalida el caché de un LP?

  • Llega un PDF nuevo asociado a ese LP (capital account, distribución, K-1)
  • Alguien edita manualmente una investment del LP en cretumdesk.com
  • Se aplica un evento de valuación (Fase 4.5) a una empresa donde el LP tiene posición
  • El usuario hace click en "Forzar regenerar"

Cómo funciona por dentro

Backend recibe: { investorId, reportType: 'direct_positions' }
↓
SQL determinístico: trae de Supabase todo lo cuantitativo
  • Datos del LP (nombre, email, código FA)
  • Todas sus investments con shares, commitment, valuaciones, MOIC
  • Calcula derivados: escenarios IPO (proporcional al mark actual), comparativa side-by-side
↓
SQL trae PDFs: SELECT raw_text FROM documents WHERE investor_id = X
  • 10-20 documentos típicamente (capital account statements, welcome letters, etc.)
  • Concatenados como contexto para Claude
↓
Claude Haiku 4.5: recibe el contexto + un schema JSON estricto 
{
  "tesis_por_posicion": { "anthropic": "...", "spacex": "..." },
  "eventos_destacados": [{ "posicion": ..., "label": ..., "valor": ... }],
  "eventos_proximos": { "anthropic": [...], "spacex": [...] },
  "resumen_performance": "..."
}
↓
Template HTML (fijo, con tu branding) recibe JSON de Claude + datos SQL → renderiza el reporte completo
↓
Frontend lo muestra en pantalla. Botón "Imprimir como PDF" usa el diálogo nativo del browser

Por qué el template fijo (y NO dejar que Claude genere todo)

  • Layout consistente: tu marca, tus colores, tu disposición. Claude no inventa layouts.
  • Números siempre correctos: los cálculos (MOIC, escenarios IPO) vienen de SQL, no de un LLM que puede equivocarse con decimales.
  • Costo mucho menor: Claude solo escribe ~5K tokens de prosa, no las 6 páginas completas. Output token count cae 5x.
  • Iteración rápida: cambiar el diseño del reporte es editar HTML, no re-tunear un prompt frágil.

Qué necesito saber para arrancar
  • ¿Cómo vinculamos cada PDF a su LP? Opciones: (a) que el script de la Fase 1 le pregunte a Claude "¿de qué LP es este doc?" al ingestar; (b) parsing del filename si Altareturn los nombra consistentemente (ej. Nathaly_Marcus_CapAccount_Q1.pdf); (c) tagging manual desde la UI. La opción (a) es más robusta pero gasta ~$1 USD por los 1000 PDFs.
  • ¿Quién mantiene las valuaciones actuales de las empresas privadas (Anthropic $900B, SpaceX $1.75T)? ¿Ya viven en investments.current_ev_b de tu tabla, o hay un proceso aparte?
  • ¿Tu coworker te puede pasar el prompt original que usó? Si ya tiene uno que funciona, lo adaptamos en vez de empezar de cero.

✓ Decisiones tomadas

1. Idioma de los PDFs → Mayormente inglés
Implicaciones técnicas: Tesseract OCR usará el modelo eng; el índice GIN de Postgres usará to_tsvector('english', raw_text) en lugar de 'spanish'. Si aparecen docs en español ocasionales, se buscan igual pero con menos precisión (sin stemming).
2. Storage del original → Dropbox + respaldo en disco duro físico
El script sube cada PDF a Dropbox /altareturn/processed/. El respaldo en disco duro lo haces tú manualmente (recomendable: copia de la carpeta entera antes de ejecutar el script, por si algo sale mal).
3. Borrar los PDFs de la compu local → Sí, automático
Después de verificar que el PDF está en Dropbox y el texto está en Supabase, el script elimina el archivo local. Si falla algún paso, el archivo se queda y se registra en el log para investigar.
4. Arrancar Fase 0 → En pausa · pendiente de revisión
El usuario está revisando cosas extra antes de dar luz verde. No se construye nada todavía — el plan queda registrado y se retoma cuando esté listo.
